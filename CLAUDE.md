## 🧠 项目设计原则

### 1. 用户优先原则
- **小白视角**：所有设计和文档从非技术用户角度思考
- **价值前置**：在用户投入任何成本前，先展示核心价值
- **信任建立**：用社会证明而非技术术语建立信任

### 2. 极简原则
- **功能聚焦**：只做启动器，不做其他
- **操作简化**：每一步都不能超过3次点击
- **认知负荷**：用户无需理解技术细节即可使用

### 3. 产品三段论
- **预测**：预测用户需求（快速启动）
- **单点击穿**：找到一个点（3秒启动）
- **All-in**：投入所有资源做到极致

## 📋 文档编写原则

### README设计心理学

#### 用户阅读路径
```
第1秒：这是什么？能帮我什么？        → 价值主张
第3秒：真的有用吗？别人在用吗？      → 社会证明  
第5秒：怎么特别？为什么比别的强？     → 核心特性
第10秒：安装复杂吗？要多久？        → 安装步骤
第30秒：怎么用？有什么注意事项？     → 使用说明
第60秒：安全吗？有问题怎么办？       → FAQ
```

#### 认知负荷管理
- ❌ 避免技术术语（CLI、bash、终端、Git、npm等）
- ❌ 避免抽象概念（上下文、依赖、环境变量）
- ✅ 使用具体场景（研究文件夹、项目文件夹）
- ✅ 使用视觉类比（超能力、智能助手）

#### 信任建立策略
1. **社会证明** - 用户数量、星级评分
2. **安全承诺** - 开源代码、不修改系统
3. **风险降低** - 可卸载、无残留文件
4. **用户评价** - 真实使用反馈

## 🔧 技术实现原则

### 代码架构
- **单一职责**：每个文件/模块只做一件事
- **依赖最小**：尽量减少外部依赖
- **容错设计**：优雅降级，不崩溃

### Git工作流程原则

#### 为什么总出错？
1. **分支保护** - main分支有规则，必须走PR流程
2. **引用冲突** - tag和分支同名导致歧义
3. **历史分叉** - rebase产生复杂冲突
4. **认知负荷** - 需要记住太多规则

#### 正确的心智模型
```
Git = 版本控制 + 协作工具

基本原则：
1. 永远在feature分支工作
2. 通过PR合并到main
3. 一次只处理一个任务
4. 冲突时优先中止而非强解
```

#### 标准工作流
```bash
# 万能公式
创建分支 → 工作修改 → 提交推送 → PR合并 → 删除分支

# 对应命令
git checkout -b feature/xxx
# ... 工作修改 ...
git add . && git commit -m "message"
git push origin feature/xxx
gh pr create && gh pr merge --squash --delete-branch
```

## 📝 质量保证原则

### 发布前必查
1. **功能测试** - 在干净环境测试安装和使用
2. **文档检查** - 按README模板逐项验证
3. **版本一致** - 所有文件版本号统一
4. **链接有效** - 所有外部链接可访问

### 常见错误模式
1. **遗漏上下文说明** - 用户不理解为什么需要
2. **步骤顺序错误** - 安装在特性前
3. **技术术语过多** - 吓退小白用户
4. **下载链接不明显** - 找不到CTA

### 预防措施
- 使用CLAUDE.md中的模板和检查清单
- 每次发布前用小白视角重读README
- 保留历史错误记录避免重犯

## 🎯 核心价值观

### 产品观
- **简单 > 复杂** - 能不加的功能就不加
- **专注 > 多功能** - 一个功能做到极致
- **用户体验 > 技术炫技** - 用户觉得简单才是简单

### 工作观
- **预防 > 补救** - 事前检查胜过事后修复
- **标准化 > 个人能力** - 流程优于依赖个人记忆
- **外部记忆 > 大脑记忆** - CLAUDE.md优于大脑记录

### 成长观
- **错误是学习机会** - 每个错误都要记录预防方法
- **用户反馈是金矿** - 痛点就是产品机会
- **持续优化 > 一次完美** - 快速迭代胜过慢速完美
